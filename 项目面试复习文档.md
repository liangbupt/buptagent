### 一、 破冰与项目全局 (项目介绍与核心难点)

**1. 你的自我介绍与项目概述。**
> **回答：** 面试官好，我是一名专注于AI Agent与后端开发的工程师。我最近主导开发了“北邮校园智能生活助手”，这是一个基于 Python、LangGraph、FastAPI 构建的多工具 Agent 平台。系统通过融合 RAG 和 MCP 技术，将原先分散在小程序的教务、生活服务聚合到了一个统一的对话大管家入口。

**2. 这个项目的出发点是什么？为什么想到做它？**
> **回答：** 纯痛点驱动。在校园里，查空闲教室、看课表、找失物招领分散在不同的小程序和公众号里，耗时且体验割裂。我想通过 LLM 的意图理解能力做一层统一的路由分发网关，目标是把获取这些服务的平均耗时缩短40%以上，真正做到“随问随拿”。

**3. 你在这个项目里承担了什么角色，主要负责什么事情？**
> **回答：** 作为项目的核心开发者主导系统架构。我主要负责了四大核心模块：第一是利用 LangGraph 确立多智能体（Supervisor-Worker）的工作流；第二是搭建基于 Chroma 与混合召回技术的白盒化 RAG 架构；第三是开发 Redis+向量库的分层级记忆体系；第四是对接 MCP (模型上下文协议) 并重构了全流底层 FastAPI 处理并发 I/O 阻塞的宕机防线。

**4. 淘天真题：选择一个我觉得有挑战的项目阐述，并描述困难点及解决方案。**
> **回答：** 该项目工程落地中核心解决了三大挑战：
> 1. **从“单点黑盒大模型”到“可靠多路路由”：** 初期大模型包揽任务易弄混接口。我引入 LangGraph 构建 Supervisor-Worker 模式，设计结构化思考链（ReAct/CoT），要求大模型必须输出 `ROUTE`、`RATIONALE` 与 `CONFIDENCE`，将低概率意图降级兜底。
> 2. **RAG 架构的“模糊检索”白盒化演进：** 纯向量检索面对“某食堂晚几点关门”等短约束经常失效。我自研了“关键词 TF-IDF + 轻量向量”混合召回，通过 RRF 融合排序，且强制携带 `source_id` 探针回传，实现 100% 溯源。
> 3. **长期记忆防污染：** 单句抱怨容易被当成长期偏好沉淀进大数据库。我设计了“Redis 短期滑动窗口 TTL + 阈值打分模型入库聚合”的双层过滤结构，极大地控制了存储膨胀和噪声。

**5. 知识库(RAG)和 MCP Server，分别解决什么问题？**
> **回答：** 
> - **知识库（RAG）：** 解决**静态的、非结构化数据**（如校园卡规定、请假流程）。依赖语义相似度。
> - **MCP Server：** 解决**动态的、结构化且须 100% 精确的数据**（如“今天下午教三某教室是否空闲”，必须查数据库）。这类不能靠大模型生成，必须查 API。

**6. 为什么偏要上一个 MCP 协议？做个单机本地调用 / 本地 Python 函数不好吗？**
> **回答：** 因为项目定位是校园能力中枢。我预期未来的数据提供方（如各大学院、后勤处）有自己的技术栈（Java/Go）。引入 MCP (Model Context Protocol) 作为标准层，可以完全解耦大模型推理端和底层业务数据端的代码逻辑，他们只需暴露 MCP Server，我这边 Agent 即插即用，不用为了迁就他们去修改工具封装代码。

**7. 这个 Agent 目前处于什么状态？只是自己使用，还是已经考虑推广？**
> **回答：** 目前主要处于内测跑通（Beta MVP 验证）阶段。系统部署在我个人的云平台上，当前主要是开放给教研室内部及熟人同学圈子限制使用，用来日常测试压榨边缘用例、收集边界 Bug 以及验证底层 Agent 分流判断的稳健性。

**8. 为什么暂时没有把它正式全校推广出去（背后的瓶颈与考量原因）？**
> **回答：** 大规模推广受压于三个现实的工程与风控考量：
> 1. **鉴权与隐私合规**：如查询“教务成绩、校园卡消费”属于绝密隐私数据。要真正打通落户，必须接入学校信息中心的统一身份认证服务 (SSO)；如果我自己截取要求用户提供校内系统的明文密码是不合规的。
> 2. **算力经济账**：每一句复杂的多意图对话，伴随在后的都是 LLM 的 3-4 次内蕴推导 (Thought)，Token消耗极其夸张。在没有将系统 Supervisor 低阶路由廉价化、部署进本地服务器推理蒸馏小模型（如本地搭建 Qwen-7B 接管简易业务流）前，大规模并发极大易刷爆账单。当前线上开放体验采用的折中策略是要求使用者挂载自定义自己的云端大模型 API Key 额度。
> 3. **交易系统的幻觉追责与风控拦截**：例如接入“食堂卡充值”、“跳蚤市场买卖”等强交易性质业务出点错就是财务事故。同时针对于 To-C 开放场景，真实校园用户很有可能会随意对机器人发送不当指令、负面情绪等。必须在系统最外层架设意图审计（针对敏感词或非法政治类提问做屏蔽下线过滤网），而端到端静默裸推则风险隐患太大。

### 二、 Agent 架构设计与工作流协作

**9. 既然有 Assistant API 现成接口，为什么还要自己用 LangGraph 手写这么多分层？**
> **回答：** 第一，数据绝对自主权，我不希望知识库和长记忆被黑盒死锁在云厂商那边；第二，细粒度的工程把控。LangGraph 是基于图状态机（StateGraph）的，它允许我随时切入任何节点（例如在工具回调前后注入自定义的审计链路、拦截敏感调用的执行等），这些全托管黑盒做不到。

**7. 单 Agent 和 Multi-Agent 你什么时候选？项目里 Agent 之间是怎么协作的？**
> **回答：** 核心取决于“任务可分解度、通信成本机制”。
> - 单 Agent：任务链路极短、垂直域（比如纯闲聊）。多 Agent 只是徒增网络时延。
> - Multi-Agent：复合长尾任务（比如“查课表->对比跳蚤市场->规划时间”）。如果塞给一个模型，容易遗忘指令、爆发幻觉。
> 
> **协作机制**：项目中采用了 **Hierarchical (Supervisor-Worker) 层级调度**。所有意图进入 Supervisor 规划并分派给下属 Worker，执行完回写到共享 StateGraph 里。

**7.1. [高频高压拷问] 如果你的每个子 Agent 仅仅只是纯粹去调一个具体的 API（查教室、查食堂），那我直接用一个基座大模型（单 Agent）挂载 3 个工具让它自己选不就行了？为什么要搞这么复杂的子 Agent 协同架构？**
> **回答：** 这是一个非常好的工程问题。看似单 Agent 挂载多工具更简单，但在真实复杂业务下会遇到三大致命瓶颈，这正是多 Agent 架构存在的必要性：
> 1. **Prompt 污染与指令超载：** 如果单 Agent 挂载校园所有工具，它的 System Prompt 会极其庞大（既要写选课规则、又要写跳蚤市场讲价逻辑）。大模型面临“上下文注意力稀释”，极易在调工具时传错参数（把选课 ID 传给查食堂接口）。拆成 Worker 后，每个子 Agent 只有**极度纯粹且特化**的 Prompt，专心负责自己的垂直领域，参数准确率大幅提升。
> 2. **错误隔离（Blast Radius）与状态机打断：** 单 Agent 是一笔到底的黑盒，如果查食堂接口挂了，它的思维链可能崩溃并胡编乱造。在 Supervisor 架构中，Worker A 查不出数据报错，仅仅是该节点打上 Error 状态返回，Supervisor 能清晰捕捉并调度其他兜底逻辑，不会影响 Worker B 的并行查课表，实现了真正的**故障降级隔离**。
> 3. **工具可扩展性与团队协作：** 以后如果接入“校园支付”或者“重型法律知识库”，它们需要专属的权限和安全网关。作为单点 Agent 无法给特定工具做权限分级。多 Agent 允许我们给敏感 Worker 节点单独配置审批流（Human-in-the-loop）和独立鉴权模型。

**8. Agent 架构选型：它是一次性调用还是循环执行的？何时停机判断逻辑是什么？**
> **回答：** 这是**基于环境反馈的条件循环执行引擎**。Supervisor 发出工具调用后，真实环境返回给它去 observe（观察），进入下一帧推导。一直循环，直到 Supervisor 发现任务圆满解决输出了 `FINISH` 标签、或者连续两次遭遇接口彻底 `Error` 报错、又或是触达系统预设的最大递归深度 `recursion_limit` 才会退出停机保护。

**8.1. [源码深挖] 你的状态机 (State Machine) 在代码层面具体是怎么设计的？各个节点之间怎么通信？**
> **回答：** 系统的通信并没有采用零散的全局变量，而是采用了极为严谨的 **Blackboard (黑板模式)**。
> 1. **状态大盘定义（强类型约束记录）：** 我使用 Python 的 `TypedDict` 定义了核心结构 `AgentState` 作为唯一数据载体。其中最关键的是 `messages: Annotated[Sequence[BaseMessage], operator.add]`，此注解代表无论图节点怎么走，新产生的对话消息都会被**增量归集拼接**在末尾，所有 Worker 皆在此池中透明查收读写。
> 2. **拓扑编排与条件导流（StateGraph）：** 代码中通过 `StateGraph(AgentState)` 将各功能端连通。利用核心的 `add_conditional_edges` 根据 Supervisor 写回在黑板中的 `next_node` 标识作为路由器决定下一个该点亮哪个 Agent。这种高低内聚的方式使得不同模型工具解耦隔离，数据层流转井井有条，调试排错时的快照抓获与追踪变得异常直观。

**9. 为什么需求不能一次性全丢给 AI，而要拆小、拆细？**
> **回答：** 一次性抛过去会导致 Attention Dilution（注意力稀释）、极易产生幻觉，并且无法控制中间错误步骤。拆解成图节点后，每一步（比如提取实体 -> API查教室 -> 总结结果）可以独立被开发人员干预验证、加入规则兜底校验。

**10. 如果用户的输入非常复杂而且混合了多个问题（如：帮我查明天有什么课，另外去问问食堂挤吗），你怎么处理？**
> **回答：** 在目前系统的代码架构中，我们利用了 LangGraph 的**状态机自动循环流转（Cyclic Graph）**机制，将复杂意图降维为串行流水线逐一击破。具体来说：
> 1. Supervisor 初次判断，先将其中的一个核心意图（如查课表）分配给 `academic_agent`。
> 2. `academic_agent` 执行完毕，将查到的课表追加到 `AgentState` 共享上下文中。而在图结构中我们写死了 `workflow.add_edge("academic_agent", "supervisor")`，这强制流程执行完后无条件流转回 Supervisor节点。
> 3. Supervisor 再次被唤醒并阅读最新上下文，发现课表查完了但“食堂状态”还没解答，于是第二次将任务路由继续发给 `life_agent`。
> 通过这种“Worker -> Supervisor”不断折返的闭环，系统就像剥洋葱一样，基于最新上下文安全地连续派发，直至所有隐性指令满足后才抛出 `FINISH` 彻底退出状态机。未来为了极致降低推理时延，再会考虑向“Prompt Array并行派发 + 顶层 Fusion节点汇总”的并发图模式改造。

**11. 系统级提示词 (System Prompt) 是怎么设计的？**
> **回答：** 采用了“角色定义设定 + 任务硬边界说明 + 结构化约束 + Few-Shot（少样本范例）”。并且加了高压防伪条款：“找不到信息就直言无法解答，回答必须携带检索 `source_id`”。

**12. Agent遇到工具频繁调用失败，或者大模型持续爆发幻觉怎么办？**
> **回答：** 系统做了三层 Fallback（降级）：
> 1. 先通过 Pydantic 捕获具体参数组装报错结构，扔给大模型要求“由于缺少xxx重试”。
> 2. 若依然失败超时，触发内部断路器熔断，跳出当前无限思考循环，切至 Mock 引擎规则化回答。
> 3. 更严重的上下文崩溃，触发 `Human-in-the-loop` 抛弃 Context 等待人工客服或手动操作介入。

**12.1. [高纬连环追问] 假如任务是拆分的（查空闲教室 -> 推荐周边火锅），如果第一步查教室大模型产生了幻觉（编造了一个假教室号），导致第二步火锅推荐跟着错。这种“幻觉累积传递”怎么破？**
> **回答：** 解决这种级联幻觉的关键在于拦截并阻断“错误上下文”向后传递。我们在代码中通过**工具层的强端点校验并回流错误 Observation** 来实现：
> 在 ReAct 工作流的 `Action -> Observation` 推理闭环中，当第一个子 Agent 拿着大模型编造的假地名去调用底层的工具 API（如 `_mcp_call` 请求真实数据库）时，后端服务因为校验不存在会直接抛出 Error 或“查无此地”。
> 我们在工具代理层会主动捕获这些异常，绝对不让系统带病向下闯，而是将其格式化为标准的 `Observation` 串（如：“工具执行失败，校区查无此坐标，请不要伪造地名重试”）返抛给当前 Agent。Agent 拿到这份受阻的观测反馈后，便无法拼装起完整的合法返回体，从根本上锁死了“把幻觉内容抛渡给 Supervisor 后游”的口子，这就彻底掐断了拿着假结果去引爆第二步火锅推荐的毒性蔓延。

**13. [扩充] Supervisor 节点如果把意图完全分发错了怎么办？(容错纠偏)**
> **回答：** 系统的容错纠偏依靠 **LangGraph 状态机的回流边（Feedback Loop）加上 Blackboard 黑板通信** 设计。因为当前系统并不像流水线一样是单向的。
> 当 Supervisor 脑子抽风把“买二手车”分发给了 `academic_agent`（学术助手）后，`academic_agent` 面临没有跳蚤市场图谱数据和工具可用的窘境。它在经历碰壁后会在 `AgentState["messages"]` 留言“抱歉，我缺乏解决二手交易的工具，无法处理”。
> 关键点在于由于代码中设定了 `workflow.add_edge("academic_agent", "supervisor")`，这个包含碰壁消息的状态会被重新传导回送给 Supervisor。Supervisor 本身被提示词约束着需要根据最新的上下文决定去留路线，这时它读到前驱失败，会在再次推理时自我纠偏，将其安全转移给 `interaction_agent`，这就构成了天然的 **Cross-Check 容错打回**。

### 三、 RAG 检索增强与数据抓取底层

**14. RAG 全流程处理细节是怎样的？**
> **回答：** 解析（目前仅支持校园结构化 JSON 数据载库） -> 切分重组（保留对应 `source_id` 映射索引） -> 特征化（利用大库 `bge-large-zh-v1.5` 切为 1024 维稠密特征） -> 写入底库（存入本地化 ChromaDB 并绑定 metadata） -> 检索期采用双路融合（自行手写的 TF-IDF 关键词计频得分跑一边，ChromaDB 的 Vector 余弦相似度跑一边） -> 尾部实施 RRF 倒数融合（1/(k+Rank) 分数叠加重排并卡定 Top K） -> 送进 System Prompt。

**15. 面对图片、PDF、Excel 表格等非结构化多格式怎么处理？**
> **回答：** 项目当前主打结构化本地轻量响应。核心事实资料基于可高度受控的本地 `campus_kb.json` 管理。这意味着我们主动舍弃了庞重易出 Bug 的 OCR 或 PDF 分解件。其实在真实校园场景落地下，核心数据字典（比如课表、餐厅索引表）本质也就是表格类 JSON 串，强制利用结构化解析映射代替粗糙无脑的 PDF 物理分割，反而是避免知识碎片化的最优选择。

**16. 你用了什么 Chunk 切分策略？非规则化文档怎么分？**
> **回答：** 我们拒绝盲目调用郎链里自带的那种死板的 `RecursiveCharacterTextSplitter` 它极易切断关键连贯语意。在我们的代码实现中，`_tokenize` 采用了更为精细贴合中文校园行文习惯的做法：针对英数混杂部分全切分打碎、针对中文汉字直接硬刚走 **Bigram (双字滑动 N-gram)** 切割组块。这种颗粒度极微且字词重叠度非常高的底层处理，保证了极细微实体名的命准。

**17. 为什么必须要加入关键词（用手写的自定义 TF-IDF），纯向量究竟差在哪？**
> **回答：** 纯向量抓取的是模糊大语义。举个例子，用户问“教三-334 教室”，但在语义空间里，“教三”加数字的组合因为分布过近会被大模型判定相似算成误召回。但我们在代码 `_keyword_score` 里直接把双字词放进循环，对 `query_tokens` 中精准存在的每个实体使用 `math.log` 的严格 TF-IDF 进行强制分数补成。这种组合极大压制了“近义但核心实体编号变异”所拉低精准率的老大难问题。

**18. 意图模糊：如输入“明天吃啥？”，RAG如何处理？**
> **回答：** 我们的底层数据 `campus_kb.json` 里，会专门存入对模糊查询的范例对齐（例如“若预算 30 ，可优先推荐..”）。当碰到毫无知识点的问询，我们会强行放开 RAG 阻断墙，利用前面 `HybridMemoryManager` 里长记忆里提点的特征喜好，去让 Supervisor 层依据 `query` 原生补全路由推理，而不完全死锁依赖于 RAG 的死知识反馈。

### 四、 分层记忆体系 (Memory Management)

**19. Agent Memory 分层记忆体系是怎么落地的？**
> **回答：** 在我的代码 `app/memory/hybrid_memory.py` 里，我自行设计并实现了一个名为 `HybridMemoryManager` 的组件，它是一套具备本地降级能力的分层体系：
> 1. **短期工作记忆 (Working Memory)：** 采用 Redis List 结构作为滑动窗口。代码中通过 `lpush` 和 `ltrim` 将最近 N 次的 `{"user", "assistant"}` 对话记录 JSON 压入缓存，并挂载 7 天的 TTL 自动过期。如果 Redis 挂了，代码底层会自动退化回内存的 `dict`。
> 2. **长期偏好记忆 (Semantic Memory)：** 针对用户长期特征（如忌口），采用 ChromaDB 建了一个 `user_long_term_memory` Collection。为了杜绝同类记忆反复灌库导致严重冗余，我在存入前利用 `abs(hash(normalize_text))` 进行本地哈希查重比对隔离，并且设置了一个 30 天自动流转移出的逻辑机制。
> 3. **全局防重问缓存 (Semantic Cache)：** 单开了一个 Chroma 空间 `semantic_cache`。专门将那些高频回答用大参数模型（BGE-large-zh）刷成余弦空间向量。当相似度超过 0.95 就会命中短路墙，直接返回之前结果，全流不过 LLM。

**19.1. [高频高压拷问] 你往这里面到底存的是什么？在你的界面上用户可能只是点点点或发些散碎问题，具体是怎么变成特征最后成为一个“向量”存储起来的？整个清洗流转的技术链路是什么？**
> **回答：** 存入原生的对话文本是毫无价值的，必须经过**“预前降重清洗 -> BGE 强语义映射 -> 元信息（Metadata）绑定”**的三步管线：
> 1. **格式化与哈希查重降噪：** 在 `hybrid_memory.py` 里，新特征（如“我不吃辣”）被存入长期库之前，我先利用 `_normalize_text` 对句子进行缩表去噪，接着通过 `_text_hash` 计算其唯一特征哈希并打上 30 天的有效期时间戳。紧接着在向 Chroma 触发 Insert 时先拿着 `user_id` 在底库里做一个精准匹配比对 `text_hash`：如果发现该长期库中已有一模一样且不过期的长期记录（防止同义内容膨胀），代码在这一步就直接 return，掐断冗余；
> 2. **稠密向量化转换（Embedding）：** 躲过清洗阻断后，系统调用了配置在本地的高阶 `HuggingFaceBgeEmbeddings("BAAI/bge-large-zh-v1.5")` 且启用了 `normalize_embeddings=True`，将提取出的纯粹事实如“去一楼自习”计算转化为 1024 维的高密向量；
> 3. **元数据绑定并 Upsert：** 最后，不仅该 1024 维长数组被写入，这个向量还会和带有 `user_id`, `created_at`, `expires_at`, 及刚刚的 `text_hash` 共 4 个字段的一条 JSON 字典被打包成一张数据表塞进 Chroma (`upsert(ids=[doc_id])`)，此后无论是 RAG 查询过滤过期时间，还是 Supervisor 做用户权限分列拉取时即可精准筛选到该个人的特定偏好特征向量。

**20. 记忆压缩与检索：随对话轮数增加，上下文暴涨怎么进行压缩？**
> **回答：** 我们并没有引入复杂的 Summarization Agent（总结代理）来增加系统负担，而是直接在底层维护了一个定长的滑动窗口。在 `HybridMemoryManager` 里，短期工作记忆由 Redis 的 `ltrim` 操作或者本地字典的长度切片（如 `[:50]`）来保障。这确保了对话上下文中永远只保留最近 N 轮的有效对话，从而在兼顾成本和响应速度的前提下，暴力规避了上下文暴涨导致的 Token 超出问题。

**21. 高频拷问：长期记忆靠大模型二次打分提取入库，高并发下引发极大的操作延迟怎么解决？**
> **回答：** 我们并没有采用消耗极大算力让大模型去后台跑 Celery 异步打分总结偏好这种激进且极易超时的做法。相反，在代码里我实现了一个基于权重的**规则评估启发式字典 (Heuristics)** `score_long_term_signal`。系统会在极短的毫秒内去排查 message 流里的用户高意图长效词（如“喜欢”、“偏好”、“不吃”、“预算”等），如果在计算后分值 > 1.0 阈值（`should_store_long_term` 评估为 True），系统才会把这部分语义交由 BGE Embedding 稠密化并插入 `ChromaDB`，这样就在主干逻辑上以极低开销实现了过滤，彻底甩掉了重度性能延迟。

### 五、 系统工程落地、高可用防线与底层并发 (后端工程八股)

**22. 系统工程与安全防线落地：工具如何做到本地化以及减少对第三方的强依附性死锁？**
> **回答：** 我们摒弃了直接在主网关里满天飞 `requests` 强连其他教务处导致单点故障的危险做法，系统引入了 MCP (Model Context Protocol) 作为解耦中枢。在 `mcp_client.py` 内部，通过 `StdioServerParameters` 将寻找教室、吃喝推荐等核心操作全部隔离调用本地 `mcp_server.py`。即便外部 API 崩溃，Agent 也只会在 MCP 通信上感知到 `ClientSession` 的返回兜底，而不会让主路由直接阻塞导致整个 uvicorn worker 耗尽宕机。

**23. CPU 飙满服务宕机：提到 FastAPI 的宕机性能问题，做过哪些排查与解决方案？**
> **回答：** 主要是出在外部 I/O 阻塞上。系统做了完全彻底的“协程流改写”。例如 `mcp_client.py` 内部全程依靠 `asyncio` 的 `ClientSession` 唤起，就连 `memory_manager` 操作 Redis 或者 Chroma 虽然有些是阻断式的，但在 HTTP 的核心对外接口 (`/chat/stream`) 我们是采用 `async def` 挂载 `astream_events` 纯异步发生器去流转事件对象的。这极大避免了被长期 I/O 卡住拖拽到 CPU 拉满假死的囧况。

**24. 数据一致性挑战：提到在 MCP 工具侧的数据聚合源，你是怎么处理的数据解耦支持测试的？**
> **回答：** 这是通过本地设计统一黑盒的中间提供者模式做平滑过渡的。我们在工具箱底层放置了一个 `ToolDataProvider`。它支持多重 `mode` 比如纯本地映射 JSON，一旦外部的测试或教务环境波动断联，系统可以瞬间降缓加载本地磁盘中的 JSON 数据。这极好地隔离了由于开发测试带来的外部数据环境脏乱差。

**25. 针对高并发真上线要求，选型里向量库有什么配置特点？**
> **回答：** 相比于大厂堆砌的重资产群集方案，本项目采用的底座依然是轻量化的 ChromaDB。虽然 Chroma 开源版不具备分布式特性，但对该本地应用而言，我们通过将其存放在持久化物理路径 `./chroma_data/` 并设置 SQLite 作为后台索引底层（结合 BGE 大模型 embedding），达成了零运维成本但完全能够吃下目前高密请求的轻量级长期检搜目标。

**26. IO 密集的大文件解析时，如何防止内存 OOM 宕机？**
> **回答：** 因为本项目直接从源头通过高度控制预处理管线生成规范化的 JSON 数据（如将长文本脱水提纯）。这就从源头上杜绝了线上实时接收、物理拆解大尺寸二进制文件（比如 `pdf.read()` 一次吃满内存）的灾难操作。所有的 IO 操作被极大局限成了对极小体积文本的序列化与特征切分，所以彻底免除了由于解析体积庞大致使服务器出现 OOM （内存溢出）的风险。

### 六、 全链路评测、模型微调与底层大模型八股机制

**27. 整体系统性能评价，你核心把控哪几类指标？**
> **回答：** 分两套关注：
> 1. 系统的业务指标盘：大模型工具路由的精确派发率（Precision），MCP 接口最终的请求业务 Success 率。
> 2. AI 平台质量面指标：**TTFT** (Time To First Token，首字吐流时延被拿来衡量体验)；**Recall Score** (RAG对所需关键字召回覆盖率) 以及不可造假的 **Faithfulness** (无幻觉严谨忠实度指标)。

**28. [AI评测落地] 评测新老系统(增加了 Prompt 或工具时)哪个好，是如何手段自动化评估（取代手动）的？**
> **回答：** 我们放弃了外采 RAGAS 这类框架，而是自己手写了一套 **LLM-as-a-Judge** 的轻量评估脚本 `evaluate_agent.py`。我们固化抽出了一份包含问题指引、预期意图与预期知识点锚点 (Expected Keywords) 的 **Golden Test Set (黄金标准答案集)**。当评估跑批时，用本地模型作为裁判去对比，不仅计算 `Recall Score`（目标意图是否按标准集合覆盖被唤起），而且打分出一个硬性评价 `Faithfulness`（输出是否忠实不编造），从而量化出系统演进时的提升或下降跨幅，使得提示词调优不再凭感觉。

**29. 检索与生成接口经常被人嫌弃耗时慢，代码实现上具体拿出了哪些耗时优化（TTFT）方案落地突破？**
> **回答：** 我们全路采用了 **SSE 流式回推（Server-Sent Events）**。在普通的 HTTP 里必须等到 RAG 和大模型串通走完结题后才推送结果。但是在我们的 `routes.py` 的 `/chat/stream` 实践中，采用了 FastAPI 的 `StreamingResponse`。
> 更进阶的一点是，为了让首字时延极短，我们通过挂载利用了 LangGraph V2 的 `astream_events` 发动机。当后台刚刚触发了 RAG 取到资料，甚至才刚刚规划好走什么意图没出回答，它就已经把携有这批数据的 Event 连带 `rag_hits` 原封打包抛向了用户设备。这意味着用户先看到了引用的片段（视觉空白极大填补），然后才是模型真正的文本 `chunk` 喷发而至。这是真正实干级的 UX 填补方案。

**30. 这几个 RAG/SFT/DPO 技术领域认知里怎么划分适用版图？**
> **回答：** 
> - **RAG (知识检索增强)**：只包治知识与常量的**事实性时效更新**缺失(新增章程变动、时刻变更等非内置预料)的查漏替换问题。 
> - **SFT (监督微调)**：治疗模型自身怎么“**端着架势**”做答和学会一些专门特异性的结构与流程语气规律(例如让其专门吐强规约束型 JSON、强制扮演校园风纪学生会等)。
> - **RLHF/DPO(强化对齐层)**：管规与价值风向收归(更具帮助亲和力与拒绝暴力违反学风的偏好倾向安全准入评断调控阀)。此外遇到小模型SFT里频频造伪“对无关语境瞎配制强结构”即是**过拟合泛滥**的状况应当挂加无相关背景预置数据的负样本结合 `Early Stopping` 策略扼杀打断。

**31. [底层原理问询] 大模型的 Temperature 指定在 0 时，回答真的是在原理物理层百分百固定死不变吗？**
> **回答：** （在忽略极其微观的异构云显卡推演浮点扰动条件下）**可以说是固定的。**因为切近到 `0` 以后触发启用了算法里叫 **贪婪解码（Greedy Decoding）**的特质，代表它会在接下来测算推吐每个 Token 标记节点的候补机率列面放弃散点几率捕捉随机只“吞”定算最可能最大的唯一值。所以在保证程序内输出不滑跑必须要将这个降停调控到底。 

**32. 模型跟服务端的底层 API（接口）中间，凭什么就能只给自然语言自动构建并拼装出符合你的参数体系并交互的？**
> **回答：** 靠着预底被厂家针对化强化洗训好的核心基建 **Function Calling（函数派调映射）**。工程手段其实是，开发层需要手撸极为精准繁重的带有各类声明变量类型的 OpenAPI 化准式 **JSON Schema** 告知；系统预留窗口传给机器，让经过大量特教调炼后知晓抓捕出字段要义的模型返照着按规范返回个字面段：机器不执行任何远程的请求代码，只能给这段“生成完毕的指令格式”，剩下的要由前端系统本身拦截反解抓取再做调用发起通信的后续交涉任务。出现偏离或者字段滑落则须利用如 `Pydantic` 将 Error 倒勾捕出灌回提示并命令“根据缺失回补改写修验重发”。

### 七、 AI 大模型底层认知与基础理论 (补充真题)

**33. 既然谈到混合检索，RRF (倒数秩融合) 算法的原理是什么？为什么比直接加权好？**
> **回答：** RRF 的公式是 \`Score = 1 / (k + Rank)\`，其中 k 通常取 60。它的核心优势是不需要关心不同检索器（如 BM25 和 Vector）返回的绝对分数量纲。BM25 的分数可能上百，而余弦相似度最多是 1。如果直接加权相加，极难调参；而 RRF 只看“排名顺序”，让综合排名靠前的文档平滑地浮到顶端，鲁棒性极强。
**33.1. [高频追问] 如果在业务中发现向量检索（语义）比关键词检索更准确，但关键词总是触发“词语漂移”拉低准确率，你该怎么调整 RRF 里的 K 值参数？**
> **回答：** 应该**适当调小 K 值**（例如从 60 降到 10-20）。在公式 `1 / (k + Rank)` 中，K 值越小，排名的倒数权重绝对差异就被放大。这能使得那路表现更好、能把正确结果排在最前列的检索器（此场景下是向量检索）占据强主导权，以此压制另一路错误召回带来的扰动。
**34. 向量库计算相似度有哪几种距离？为什么通常推荐用余弦相似度（Cosine Similarity）？**
> **回答：** 常见的有欧氏距离 (L2)、点积 (Dot Product) 和余弦相似度 (Cosine)。我们大多采用余弦相似度，因为它对文本的“绝对长度”不敏感。比如一段长文本和一段短文本，如果在多维空间中的“方向”一致，说明语义一致，余弦相似度高。如果用欧氏距离，长文本词频多导致模长很大，距离就会被拉远，从而误判语义不匹配。

**35. 开发 Agent 时，选择 LangGraph 的核心考量是什么？与原生 LangChain Agent有何区别？**
> **回答：** 原生的 LangChain \`AgentExecutor\` 本质是一个封装好的黑盒 while 循环，很难调试“到底它在第几步做错了决策”。LangGraph 引入了 StateGraph（状态图）的概念，将 Agent 的思考、调用工具定义为 Node（节点），用 Conditional Edges（条件边）做流转判断。任何一步执行完都会回写状态，这让支持并发、多 Agent 层级调度以及强制流打断（Human-in-the-loop）变得可编程、可观测。

**36. Prompt 工程里的 CoT (思维链) 和 ReAct 模式在 Agent 中是如何发挥作用的？**
> **回答：** 
> - **CoT (Chain of Thought)：** 核心是 “Let's think step by step”。强迫模型在给出最终答案前，先吐出中间的推导过程，能够大幅降低复杂逻辑问题的幻觉率。
> - **ReAct (Reason + Act)：** 是 Agent 的灵魂。它要求模型输出 \`Thought\`（思考当前该做什么）、\`Action\`（决定调用什么工具及参数）、\`Observe\`（拿到工具结果后的观察）。项目中基于 LangGraph 的节点循环，完美复盘了 ReAct 的这种交替推进范式。

**37. 大语言模型的本质是什么？为什么会出现幻觉（Hallucination）？**
> **回答：** LLM 的物理本质是 **Next-Token Predictor（下一个标记预测器）**，基于统计学概率进行文字接龙，本身没有真正的“世界常识记忆”或“逻辑交叉验证”机制。在缺乏强上下文约束、或知识盲区内，系统依然倾向于补全具备高概率通顺但毫无事实根据的连贯句法，这就产生了幻觉。我们可以通过 RAG 输入事实、调低 Temperature，强制输出 JSON 结构等手段来最大程度压制幻觉。

### 八、 研发效能与 AI 辅助编程实战 (Web Coding)

**38. 在这个项目（甚至所有项目）推进时，你是如何深度使用 AI（如 Web Coding / Cursor 等）辅助编程的？**
> **回答：** 我总结了一套非常标准化的 Workflow（AI 规划 - 执行范式）：
> 1. **全盘架构与文档拉齐**：绝不一上来就抛一句泛化的功能让 AI 乱写。我先将它设定为产品经理角色，将脑海里的模块目标喂给它互相讨论拆解，最后勒令它整理产出固定的 `Markdown` 规范级大纲文档（SRS 需求映射）。
> 2. **最小试错跑通 (MVP)**：对着文档要求它按最简陋的核心主骨干，仅实现一个可跑通的基础请求 Demo（例如只通 1 根 HTTP 测试链接），确认大方向不歪。
> 3. **层层挂接扩充**：在 MVP 上叠加扩展逻辑，每新增一块（如接入 MCP、写好日志缓存），同时命令它配套产出一个对应的 Test Script (单测用例) 保证增量可用无破坏。

**39. 让 AI 大段改写长文件时，它极容易跑飞发生“上下文丢失”、“越改越乱”，这种批量长任务痛结你是怎么规避治愈的？**
> **回答：** 工程经验有两点：
> 1. 编辑器记忆清理与降维压缩：及时利用 `Compact` 等清除过期命令或对无效的拉锯探索对白做信息定本重排。保持会话历史永远紧凑。
> 2. 很多高端的代码助手自带的智能内核底层其实会有“多源总结摘要压缩存储”机制；配合强制让 AI 把重大架构决策记录到硬盘实体文件中作为“硬挂载文档点（Anchor）”，相当于让它一直揣着那三五条不变的戒律（Prompt Skill）。这会让大模型在几十轮长征聊天后仍能锁住主心骨目标。
