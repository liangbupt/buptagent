# 🚀 北邮校园 Agent 项目｜高阶面试拷问与复习指南

本文档按**STAR 法则**与**高压拷问**场景深度重构，专为高级研发/架构面试准备。

---

## 1. 核心 STAR 故事线（建议作为自我介绍与深入解答的主干）

### 🌟 场景 1：从“单点黑盒大模型”到“可靠多路路由”的架构演进
- **[S] 情境**：初期让大模型包揽所有任务，常常弄混“教务查询”与“跳蚤市场”的底层工具，上下文超载严重，报错难以追溯。
- **[T] 任务**：实现一个高伸缩、低耦合、高可观测的任务分发流。
- **[A] 行动**：引入 LangGraph 构建 Supervisor-Worker 模式。我设计了结构化思考链（CoT），要求 Supervisor 强制输出 `ROUTE`、`RATIONALE`和`CONFIDENCE`；同时为防止错误阻断，将低概率意图进行降级兜底回退计算。
- **[R] 结果**：工具误调率显著下降，所有路由分发逻辑实现了系统级落库审计，排错不再需要重放 prompt（通过 `/api/audit` 可直接查询当时环境决策链）。

### 🌟 场景 2：RAG 从“模糊检索”到“白盒可解释架构”
- **[S] 情境**：单纯向量检索在处理诸如“某食堂晚几点关门”这种短约束条件时经常“找偏”（只单纯匹配了食堂二字）。
- **[T] 任务**：提高短文本与局部长尾查询下的 RAG 召回精度，同时在发生幻觉时能快速追责数据源。
- **[A] 行动**：自研了混合召回模块（Hybrid RAG）。将原本单路替换为“TF-IDF 关键词 + 轻量级向量”召回，后接 RRF 排序融合，再用 token overlap 兜底重排。并且**每个召回片段强制携带四维得分和 source_id 回传到上游**。
- **[R] 结果**：不再有“死无对证”的业务异常；不仅解决了强约束条件召回漏缺问题，而且每一次 AI 给出的结果后边都附挂准确片段，在真实演示和答辩时提供了强证据支撑。

### 🌟 场景 3：长期记忆的防污染机制
- **[S] 情境**：用户的历史倾向很容易因为临时反悔或单句抱怨（如“这次食堂真慢”），被系统误认为长期偏好沉淀进大数据库，形成噪声积累。
- **[T] 任务**：实现一个高保真、低重复、可遗忘的长效记忆引擎。
- **[A] 行动**：第一层加 TTL，用 Redis 和 Chroma 设定差异化的时效性。第二层砍掉了完全依靠规则触发的记录机制，引入微调打分阈值模型和文本哈希去重法。
- **[R] 结果**：极大地降低了数据库持久化膨胀速度，有效过滤了无关噪声。同时开放用户级别记忆擦除接口，迎合了当代系统的隐私合规标准。

## 2. 高压拷问（压力面防守话术）

**🔥 高压拷问 1：既然有 Assistant API 各种现成接口，你为什么还要自己用 LangGraph 手写这么多分层？**
> **防守反击策略（强调控制力、解耦与数据主权）：**
> “Assistant API 确实快，但这项目定位是校园中枢。第一，我需要绝对的数据自主权，不能把知识库源文件和长期库彻底黑盒化锁死在厂商云端；第二，为了精细化风控，LangGraph 允许我介入任何子图节点（比如我在工具回调前后注入自定义审计链路，甚至进行动态敏感拦截），这些都是黑盒全托管做不到的。”

**🔥 高压拷问 2：你的长期记忆靠再次打分提取，如果高并发场景每次请求都调大模型去打分，这延迟和成本你能接受吗？**
> **防守反击策略（承认现况并提供高可用演进思路）：**
> “您说得很对，当前为了隔离噪声牺牲了一定 QPS 吞吐。目前的架构是原型实验；在工业落地阶段，我会把【写记忆】动作从用户主链路完全**解耦到后台异步队列（如 Celery/Kafka）**去慢慢跑打分和落库。对话核心的强弱网链路上只做记忆读取，从而保证响应主体的 P95 延迟。”

**🔥 高压拷问 3：为什么偏要上一个 MCP 协议？这又不是大生态，做个本机本地的类函数反射调用不好吗？脱裤子放屁？**
> **防守反击策略（强调面向未来的协议级标准化解耦）：**
> “因为项目定位是能力枢纽，我预期【能力提供方】和【能力分发方】不在同一研发栈下。接入 MCP，可以使前端推理与 Agent 模型成为大脑，而各个独立二级学院、后勤部门自己去用 Java 乃至 Go 去部署 MCP Server 提供数据即可，我无需为了迁就他们去修改工具封装代码。这虽然前置学习成本高，但具备业务上的跨组织无缝集成意义。”
